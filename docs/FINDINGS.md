## 1. 디스크 구조 (US/JP 공통)

ISO9660 루트:
- `SCUS_942.54` (US) / `SCPS_100.59` (JP): PSX-EXE 실행 파일, LBA 24
- `PROT.DAT`: 메인 데이터 아카이브, US LBA 242 / JP LBA 246
- `DMY.DAT` (37MB), `XA`, `MOV`, `SYSTEM.CNF`, `CDNAME.TXT`

### 실행 파일 헤더
- **US**: load `0x80010000`, entry `0x80026C28`, size `0x6B800`
- **JP**: load `0x80010000`, entry `0x80027B20`, size `0x6D800`
- **CN**: load `0x800AA000`, entry `0x800AB1B0`, size `0x47000` ← **완전히 새 로더**
  - CN 실행파일 `0x8800~0x38800`은 **압축 데이터**(엔트로피 7.75, 방식 미상 — 게임 LZSS 아님)
  - 여기에 CN의 폰트 스트리밍 엔진 + 원본 게임코드가 압축돼 있을 것으로 추정

## 2. PROT.DAT 아카이브

```
u32 [0] = 0 (version)
u32 [1] = 파일 수 (JP 1235 / US 1235)
u32 [2..] = 섹터 오프셋 (offset << 11 = 바이트 주소)
파일 i = 섹터 [offs[i], offs[i+1])
```
파서: `tools/prot.py`, `tools/legaia.py:prot_files()`

### 파일 안의 PACK 컨테이너
많은 PROT 파일은 PACK 구조:
```
u32 fileCount
u32 totalDecLength
per entry: u24 decLength, u8 fileId, u32 compAddress (pack 시작 기준)
각 entry는 LZSS 압축됨
```
파서: `tools/legaia.py:pack_parse()`

**맵 파일 fileId 규약** (95개 맵 공통):
- fileId 1: ~266KB (3D/맵 데이터)
- fileId 2: ~187KB
- **fileId 3: 텍스트/스크립트** ← 번역 대상
- fileId 5: ~55KB
- fileId 7: 인덱스 (~720B)

## 3. LZSS 압축 (게임 내부)

`tools/legaia.py:lzss_decompress()`, C판 `tools/lzss_fast.c` (compress+decompress, 라운드트립 검증됨)

- 링버퍼 4096, 쓰기 시작 위치 **4078 (0xFEE)**
- 플래그 비트 LSB-first
- bit=1 → 리터럴 1바이트
- bit=0 → u16 리틀엔디안: `len = ((v>>8)&0xF)+3`, `pos = ((v&0xF000)>>4)|(v&0xFF)`
- **종료 조건 = 해제 후 크기(decLength)** (압축 입력 크기 아님 — 중요)

출처: JP `SCPS_100.59` @ RAM `0x8001A15C` ("decode" printf 하는 함수), 그리고
LegaiaText 레퍼런스 툴 소스와 일치 확인.

## 4. 스크립트 구조 (fileId 3 해제 후)

```
0x00..0x22 : 헤더
0x22 u16 count0, 0x24 u16 count1, 0x26 u16 count2  (count = 합)
0x28 u24 : footer offset
0x2B..    : u24 포인터 테이블 (count개)
base = 0x2B + count*3
각 스크립트 = data[base+ptr[i] : base+ptr[i+1]]
```
파서: `tools/legaia.py:script_parse()`

### 텍스트 인코딩
- **JP**: 원시 Shift-JIS 2바이트
- **US**: 평문 ASCII (제어코드 섞임)
- **CN**: ASCII 자리에 2바이트 코드(선두 ≥0x80) 삽입, 구두점·제어코드 유지

### 제어코드 (US/JP)
- `0x1F` 텍스트 런 시작, `0x00` 종료
- `0x0A 0x0B 0x0D 0x10 0x12 0x16` : 줄바꿈류
- `0xC1 0x00` : 플레이어 이름 삽입
- `0x7F` : 패딩 (CN이 남는 자리 채움)
- 메시지 종료 마커: `$`(0x24), `&`(0x26)

## 5. 폰트 (핵심)

### 위치
- PROT.DAT **파일 0** 안의 TIM
- JP: TIM 헤더 `0x7F34`, 픽셀 `0x7F74`
- US: TIM 헤더 `0x7F40`, 픽셀 `0x7F80`
- (사용자 에디터 파일0-상대 오프셋 `0x6780` = PROT절대 `0x7F80` = US 폰트)
- 크기 256×256 4bpp, VRAM 목적지 **(896, 0)**, tpage `0x0E`
- 두 번째 폰트 페이지 TIM `0x11218` → VRAM (960,256)

### 듀얼 레이어 (JP/CN 공통, US 원본은 단일)
- 한 4bpp 픽셀 니블 = `(레이어B << 2) | 레이어A`
- **레이어A = 니블 하위 2비트**, **레이어B = 니블 상위 2비트** (각 3계조: 값 0/2/3 또는 0/1/2)
- CN 패치: 레이어B(상위 2비트)에 간체자를 그려넣음 → 렌더링으로 선명하게 확인됨
- US 원본: 니블 값 {0,1,2,14,15}만 사용 (단일 레이어, ASCII만)
- CN 패치: 니블 16값 전부 사용 (듀얼 레이어)

### 팔레트가 레이어를 선택
- TIM `0x11218`의 CLUT = **16색 × 16팔레트**, VRAM (0,510) 로드
- CLUT id `0x7F80`~`0x7F83` = 서로 다른 팔레트
  - 팔레트0(0x7F80): 밝은 인덱스 {1,2,9,10,12,14}
  - 팔레트1(0x7F81): {1,2,4}
  - 팔레트2(0x7F82): {1,3,14,15}
  - 팔레트3(0x7F83): {1,2,3}
- 같은 폰트 픽셀 + 다른 CLUT = 다른 레이어/색으로 렌더

### JP 문자표 (US와 다름)
- JP: `SCPS_100.59` 파일오프셋 `0x65F80` (RAM `0x80075780`), **776개** BE SJIS 코드
  - 순서: 기호 → 영숫자 → 가나 → 한자(가나다순), `腕`으로 끝
  - `data/jp_charset_776.txt`
  - 셀 배치: 20열, 12×12px, 60px 블록4개(스트라이드 64px)
  - 레이어1 = 인덱스 0~419, 레이어2 = 420~775
  - ⚠️ 이 배치로 PoC를 만들었으나 게임 화면에서 글자가 어긋남 → JP 배치 규칙이 아직 부정확

### US 폰트 배치 (확정)
- ASCII: **셀 = ASCII − 0x20**, 16열, **16×16px 셀** (첫 두 행 `、。．・…` / `0-9A-E` 시각 검증 완료)
- 소형 HUD 폰트(8×8)는 별도: 렌더러 `0x8001A374`, 문자열 처리 `0x8001A078`
  - 예: `0-7`→U=(c-0x30)*8,V=0 / `8-9`→V=8 / `a-h`→V=0x10 ...

## 6. US 폰트 렌더 파이프라인 (디스어셈블 결과)

- `0x8001A078` : 문자열 렌더러 (printf류, `%`포맷·`\n`처리, gp+0x664=X gp+0x668=Y)
- `0x8001A374` : 문자 1개 그리기 (8×8 sprite, GP0 0x74), CLUT `0x7F80` 하드코딩
- `0x8001AD38` : sprite 프리미티브 생성 (a0=x a1=y a2=U a3=V, stack=CLUT)
- ⚠️ **대사창 16×16 렌더러는 아직 미확정** — CLUT을 변수에서 읽어 상수검색으로 안 잡힘
- CN이 2바이트 처리를 넣은 코드는 CN 새 로더(압축)에 있어 US 원본엔 없음

텍스트 버퍼에 브레이크포인트 → 대사 렌더러 PC 확보 → 거기서 훅 지점 확정.

## 7. 한글화 전략 (US 베이스)

CN이 증명한 방식을 참고하되 우리 규칙으로:
1. 텍스트는 평문 ASCII → 2바이트 코드 확장 (선두 ≥0x80)
2. 폰트 페이지 듀얼레이어 활용 → 용량 ~1058셀 (JP 12px 기준) 또는 US 16px 기준 재계산 필요
3. 한글은 조합형(자모 ~70) 또는 빈출 음절(~1000)로 축소 → 1페이지에 수용 가능성
4. 대사 렌더러에 2바이트 디스패치 ASM 훅 필요 (지점 미확정 → 디버거로 해결)

### 검증된 빌드 파이프라인
- `tools/build_poc2.py` : 대사 교체 → 스크립트 재구성 → PACK 재압축 → PROT 기록
- `tools/write_bin.py` : PROT → Mode2/2352 BIN, EDC/ECC 재계산
- 라운드트립·무결성 전부 검증됨. (단 폰트 매핑 미확정으로 최종 화면출력은 미완)
