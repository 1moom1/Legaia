# 🔴 절대 건드리면 안 되는 것 (PROT.DAT)

**전부 실제로 건드렸다가 게임이 깨진 구간이다.**
빌더는 `tools/protected.py` 의 `assert_protected()` 로 반드시 검증할 것.

## 1. 숫자 폰트 — `0x015C38 ~ 0x019438` (14,336 B)

TIM `0x11218` (VRAM 960,256) 의 **V144~255** 영역.
**메뉴의 HP / MP / LV / G / TIME 숫자를 그리는 폰트다.**
지우면 숫자가 전부 사라진다.

```
TIM 0x011218 (256x256 4bpp, VRAM 960,256)
  CLUT  0x01122C (512B)      ← 대사 팔레트. 잉크값(3) 한 칸만 수정 허용
  IMG   0x01142C
    ix @ 0x011430 = 960      ★변경 금지 (옮기면 숫자 UV 가 전부 깨짐)
    iy @ 0x011432 = 256      ★변경 금지
  픽셀  0x011438 ~ 0x019438  (32,768B, 128B/행)

  픽셀 오프셋 = 0x011438 + V * 128

  V   0~ 79 : 0x011438 ~ 0x013C38   개발 메모   → ✅ 사용 가능 (페이지3)
  V  80~143 : 0x013C38 ~ 0x015C38   빈 영역     → ✅ 사용 가능 (페이지2)
  V 144~255 : 0x015C38 ~ 0x019438   🔴 숫자 폰트 → ❌ 절대 금지
```

⚠️ 이 TIM 은 겉보기에 "개발용 디버그 폰트"처럼 보인다
(일본어 메모 "ここは常駐エフェクトが入る予定 Pochi" 가 섞여 있음).
**속으면 안 된다. V144 아래는 게임이 실제로 쓰는 폰트다.**
CN 패치도 이 TIM 은 US 와 100% 동일하게 두었다.

## 2. 팔레트 — 잉크값(3) 한 칸만 수정

CLUT `0x01122C` (16색 × 16행).
row7,8,9 는 원본에 색이 꽉 차 있고 **게임 UI 가 쓴다.**
통째로 덮으면 메뉴가 깨진다. → **값3(한글 잉크) 한 칸만** 바꿀 것.

또한 **다른 팔레트의 값3 을 투명화하지 말 것.**
UI 폰트 TIM(`0x18E0`)이 값3 을 2072px 쓰고 있어 숫자가 사라진다.
(투명화는 애초에 불필요 — 한글/ASCII 는 V 영역이 분리되어 있다)

## 3. 디스크 구조

`XA` 는 빈 디렉토리가 **아니다** — 오디오 34개(~160MB) 를 담고 있다.
`MOV` 도 동영상 6개. **디스크 실제 빈 공간은 2섹터뿐.**
→ PROT.DAT 확장은 불가능. TIM 추가도 불가능.

## 2. PROT.DAT → PACK → LZSS → Script

```
PROT: u32[0]=0, u32[1]=파일수(1235), u32[2..]=섹터오프셋 (offset<<11)
PACK: u32 fileCount, u32 totalDecLength, per entry: u24 decLength, u8 fileId, u32 compAddress
Script: 0x22 u16 count0/1/2, 0x28 u24 footer, 0x2B.. u24 포인터테이블, base=0x2B+count*3
```
맵 파일 **fileId 3 = 텍스트/스크립트**

**LZSS**: 링버퍼 4096, 시작 4078(0xFEE), 플래그 LSB-first,
bit=1 리터럴 / bit=0 u16LE `len=((v>>8)&0xF)+3, pos=((v&0xF000)>>4)|(v&0xFF)`,
**종료조건 = 해제후 크기**

## 3. 텍스트 (US = 평문 ASCII)

```
<1F> 텍스트런 시작 ... <00> 종료
예: <1F>The Memory Statue will<00><1F>remember things for you...<00>
```

| 코드 | 의미 |
|------|------|
| `0x00` | 문자열 종료 |
| `0x7C` | 제어 (1바이트) |
| `0xCE <n>` | 테이블 조회 삽입 (`0x80074050 + n*4`) |
| `0xCF <n>` | **팔레트 전환** → CLUT = n + 0x7F86 |
| 그 외 | **전부 글리프로 렌더** |

**매크로**: `0xC1`=플레이어이름(2482회), `0xC2`=아이템명(457), `0xAC`(503), `0xC7`(454)

## 4. 🔑 대사 렌더러 = `0x80036888`

**찾은 방법**: PCSX-Redux에서 텍스트버퍼 `0x800740EC` 에 Read 브레이크포인트.

```
텍스트 버퍼 = 0x800740EC   (하드코딩)

문자 c 렌더:
    U = (c & 0x0F) * 16
    V = (c & 0xF0) - 0x20
    글리프 14 x 15 px, GP0 0x64 스프라이트

CLUT = [0x8007B454](u16) + 0x7F86     ← 팔레트가 "변수"
    0xCF <n>  →  0x8007B454 = n

폭 = [0x80073F1C + c]                  ← 가변폭 테이블
    EXE 파일오프셋 0x6471C, 디스크 LBA 224
X += 폭 + 1
```

**배제된 후보**: `0x8001A374`(8x8 HUD폰트), `0x8001A078`(printf류),
`0x8001ABC8`/`0x8001C7A0`(숫자), `0x8001BE80`(3D정점)

## 5. 폰트

- PROT.DAT 파일0 내 TIM: 헤더 `0x7F40`, **픽셀 `0x7F80`**, 256×256 4bpp
- VRAM (896,0), tpage 0x0E
- 타일에디터로 열 때: 파일0-상대 오프셋 `0x6780`
- ASCII 배치: 16열 격자, U/V는 위 공식대로

### 니블 값 (핵심)
- **US 원본은 값 {0,1,2,14,15} 만 사용** → 잉크 = **14, 15**
- **224개 셀 전부에서 값 1/2/3 이 완전히 비어 있음** (검증됨)
- CN 패치는 16값 전부 사용 (듀얼레이어)

### 팔레트
- TIM `0x11218` 의 CLUT = 16색 × **16팔레트**, VRAM (0,510)
- **PROT 오프셋 `0x1122C`**
- CLUT id = `arg + 0x7F86`, 팔레트 row = `arg + 6`
- 대사가 쓰는 arg: 0(기본), 4,5,6,7,9 (색 강조)
- **arg 1,2,3,8 (row 7,8,9,14) 미사용** → 한글용으로 재정의 가능

## 6. ✅ (A)방식 한글 출력 — 성공

**ASM 훅 불필요.** 폰트 + 팔레트 + 텍스트 + 폭테이블만 수정.

1. **한글을 니블 값 1/2 로** 폰트에 그림 (ASCII는 14/15 → 충돌 없음)
2. **팔레트 row 7** (`0xCF 01`) 을 한글 전용으로:
   값1=흰색, 값2=회색, **값14/15=투명**
3. **기존 사용 팔레트**(row 6,10,11,12,13,15)의 값1/2/3 → **투명화**
   (안 하면 한글이 영어 대사에도 나타남)
4. 텍스트: `0xCF 01` + 한글코드 + `0xCF 00`
   **길이를 원문과 동일하게** → 포인터 재계산 불필요
5. **폭 테이블 패치 필수**: 한글 코드 폭 → 13px
   ⚠️ 안 하면 글자가 겹침 (v1 실패 원인)

빌드: `tools/build_us_kr2.py` → `tools/write_us.py`

## 7. ⚠️ 코드 공간 (현재 병목) — 1바이트의 벽

**하드 리밋**: 코드 `c` → `V = (c & 0xF0) - 0x20`
- 상위니블 0x0, 0x1 → V 음수 → **사용 불가**
- 유효 상위니블 = 0x2~0xF (14행) × 16열 = **224칸이 절대 한계**

**그중 실제 안전한 코드**:
- 영어가 ASCII(0x21~0x7E) 대부분 사용
- 0x80+ 는 매크로/제어로 119종 사용 (0xC1=이름 3517회, 0xC2=아이템 611회 등)
- **전체 맵 기준 완전 안전 = 4개** (`91 DB E3 F5`)
- **단일 맵(map6) 기준 = 72개** ← PoC는 이걸로 성공

**실전 번역 규모**: 영어 대사 25,272개 런 / 약 495,743자
→ 한글 고유 음절 **700~1000자** 필요 → **1바이트로는 절대 불가**

## 8. ✅ 2바이트 훅 — 성공 (게임 화면 확인)

**"기억의 상은 / 당신을 기억합니다" 출력 성공.** 224칸 한계 돌파.

### 🔑 결정적 발견: 게임에 이미 2바이트 문자 체계가 있다

대사 렌더러(`0x80036888`)는 그리기 전에 **텍스트 전처리 함수 `0x80036514`** 를 호출한다.
```
0x800368E8: jal 0x80036514      ; a0=0x800740EC(출력버퍼), a1=원본텍스트, a2=1
```
이 함수가 원본 텍스트를 `0x800740EC` 버퍼로 복사하며 매크로를 확장하는데,
**`(byte & 0xF0) == 0xC0` 인 바이트를 2바이트 문자로 처리**한다.

| 바이트 | 전처리 동작 |
|--------|-------------|
| `0xCF` | 2바이트 복사 (팔레트) |
| `0xCE` | 2바이트 복사 |
| `0xC1~0xC7` | 매크로(이름/아이템) → 점프테이블로 확장 |
| `< 0x08` | 1바이트 (제어) |
| 그 외 | 1바이트 |

⚠️ 이 범위 **밖** 의 마커(예 `0x92`)를 쓰면 전처리가 1바이트로 쪼개고,
뒤의 인덱스 바이트를 제어문자로 오처리 → **렌더러 도달 전에 텍스트가 깨짐**.

### 채택한 인코딩: `0xCE <n>`
- `0xCE` 는 전처리를 2바이트로 자동 통과
- `0xCE` 원래 기능(테이블조회)은 대사에서 **13회만** 사용, 인자값 `{02,03,0B,0E,21,80}` 뿐
- → **`n >= 0x90` 이면 한글**, `n < 0x90` 이면 원래 기능 (안전한 분리)

### 훅 구조 (`tools/hook3.py`)
훅 자리 `0x8007AC00`, 훅 지점 `0x80036960` 을 `j 0x8007AC00` 로 교체.
```
  addiu t0, zero, 0xCE
  bne   v1, t0, L_glyph        ; v1 != 0xCE -> 글리프
  lbu   t1, 1(s3)              ; 인덱스
  addiu t0, zero, 0x90
  sltu  t2, t1, t0
  bnez  t2, L_ce               ; n < 0x90 -> 원래 CE 기능
  j     L_hangul
L_ce:    j 0x80036968
L_glyph: j 0x80036B24
L_hangul:
  addiu t1, t1, -0x90          ; idx
  ... 프리미티브 필드 채우기 (U/V 는 음절식) ...
  U = (idx & 0x0F) * 16
  V = 144 + (idx >> 4) * 15
  move  a1, s0                 ; 등록 인자
  addiu s3, s3, 1              ; 원본이 +1 더 -> 총 2바이트 소비
  j     0x80036B88             ; ★ 원본 등록/전진 코드 재활용
```

**핵심 설계**: 등록(`jal 0x8003d2c4`)과 X전진을 훅에서 직접 하지 말고
**원본 `0x80036B88` 로 점프**해 맡긴다. (직접 jal 하면 크래시)
원본이 `[s3]`(=인덱스바이트)로 폭을 조회하므로,
**폭테이블 `0x6471C + 인덱스바이트` = 13** 으로 패치 → X 전진 = 14px.

## 8-B. ⚠️ 훅 작성 시 함정 (전부 실제로 겪음)

### 1. 딜레이 슬롯 — 분기 다음 명령은 항상 실행된다
`0x80036960` 을 `j 훅` 으로 바꿔도 `0x80036964`(`addiu v0,zero,0x20`)는 **여전히 실행**된다.
→ 훅 진입 시 이미 `v0 = 0x20`. `v0` 가 0xCE 라고 가정하면 안 됨.
→ 훅에서 `t0` 에 0xCE 를 직접 로드해 비교할 것.
(글리프 경로 `0x80036B24` 는 `v0 == 0x20` 을 공백 판별에 쓰므로 이 값이 필요하기도 하다)

### 2. 🔴 로드 지연 슬롯 (R3000A) — 가장 찾기 어려웠던 버그
`lw`/`lhu`/`lbu` **바로 다음 명령**에서 그 레지스터를 읽으면 **갱신 전 값**을 읽는다.
```
  lhu   v1, -0x4bac(v0)     ; v1 = [0x8007B454]
  addiu v1, v1, 0x7f86      ; ✗ 아직 갱신 전! 이전 v1 값이 쓰임
```
증상: CLUT 에 쓰레기값(0x0006) → 한글이 **아예 안 보임** (크래시도 없음).
디버거에서 `v1 = 0x64810006` 이 단서였다 (0x64810006 - 0x7F86 = 0x64808080 = 직전 color 값).

원본 코드는 사이에 2명령을 끼워 이를 회피한다:
```
0x80036B74: lhu   v1, -0x4bac(v0)
0x80036B78: addiu v0, zero, 0xf     ; 지연 흡수
0x80036B7C: sh    v0, 0x12(s0)      ; 지연 흡수
0x80036B80: addiu v1, v1, 0x7f86    ; 이제 안전
```
→ 훅에서도 로드 후 `nop` 을 넣을 것.

### 3. 텍스트 인코딩: `0x00` 은 문자열 종료
- 음절 인덱스 `0x00` 금지 (`0xCE 0x00` → 종료로 오인)
- **끝에 `0xCF 00`(팔레트 복귀)을 두면 안 됨**:
  전처리가 `0xCF` 를 2바이트로 처리하며 뒤의 `0x00`(종료마커)을 인자로 삼켜
  문자열 끝을 못 찾고 **버퍼 오버런** → 대사 통째로 안 나옴.

### 4. 프리미티브 등록은 직접 하지 말 것
훅에서 `jal 0x8003d2c4` 를 직접 호출하면 크래시(훅 중간에서 실행 정지).
원본 `0x80036B88` 로 점프해 등록/전진을 맡기는 것이 안전.

## 8-C. 음절 폰트 배치 (현재)

ASCII 폰트 페이지 `(896,0)` 256×256 안의 **빈 영역** 사용:
- V 0~143 (코드 0x20~0xAF): 기존 ASCII/매크로 글리프
- **V 144~255: 완전히 비어 있음** → 음절 배치
- `U = (idx & 0x0F) * 16`, `V = 144 + (idx >> 4) * 15`
- 16열 × 7행 = **약 110 음절** (tpage 전환 불필요)

⚠️ 실전(700~1000 음절)엔 부족 → **tpage 전환**으로 다중 페이지 필요.
→ ✅ 8-D 이후 완료. 최종적으로 8-S2 에서 한자 TIM 2페이지까지 확장해 1,100칸.

## 8-D. ✅ 다중 폰트 페이지 (tpage 전환) — 성공

**Galmuri11 12×12 픽셀폰트 + tpage 전환으로 224 음절.** 게임 화면 검증 완료
(한글 출력 + 메뉴 숫자 정상).

### 폰트: Galmuri11 (픽셀폰트)
- `size=12` 에서 **안티앨리어싱 0%**, 잉크높이 11px, 글자폭 12px — 픽셀 그리드에 완벽 정렬
- 완성형 한글 11,172자 전부 지원
- Unifont 축소본보다 훨씬 선명 → **채택**
- 글리프 12×12, 잉크 = **니블값 3** (ASCII 폰트가 안 쓰는 값)

### VRAM / TIM 지도 (⚠️ 매우 중요)

| TIM | VRAM | tpage | 내용 | 한글 사용 |
|---|---|---|---|---|
| `0x7F40` | (896,0) 256×256 | `0x000E` | ASCII 폰트 | **V144~255** (페이지1, 144칸) |
| `0x11218` | (960,256) 256×256 | `0x001F` | V0~79 개발메모 / V80~143 빈칸 / **V144~255 숫자폰트** | **V80~143** (페이지2, 80칸)<br>**V0~79** (페이지3, 96칸) |
| `0x18E0` | (896,256) | `0x001E` | UI 폰트 (니블값 3 을 2072px 사용!) | ✗ |

**총 320 음절** (페이지1 144 + 페이지2 80 + 페이지3 96)

### 🔴 절대 하지 말 것 (전부 실제로 겪음)

1. **TIM `0x11218` 의 V144~255 를 건드리지 말 것 — 숫자 폰트다.**
   지우거나 VRAM 좌표를 옮기면 **메뉴의 HP/MP/G/TIME 숫자가 통째로 사라진다.**
   (겉보기엔 "개발용 디버그 폰트"처럼 보이지만 — 일본어 메모 "ここは常駐エフェクトが入る予定"가 함께 있음 — 실제로는 게임이 쓰는 폰트다)
   쓸 수 있는 건 **V80~143 (64px) 뿐**.

2. **팔레트를 통째로 덮지 말 것.** row7,8,9 는 원본에 색이 꽉 차 있고 게임 UI 가 쓴다.
   → **한글 잉크값(3) 한 칸만** 수정. 나머지 15색은 원본 보존.

3. **다른 팔레트의 값3 을 투명화하지 말 것.**
   UI 폰트 TIM(`0x18E0`)이 값3 을 2072px 쓰고 있어 숫자가 사라진다.
   그리고 투명화는 **애초에 불필요**: 한글 글리프는 V144~255 / V80~143 에만 있고
   영어 대사는 U/V 가 ASCII 영역(V0~143)만 가리키므로 한글이 영어에 나타날 수 없다.

### 중국어 패치 참고 (CN PROT 분석)
- CN 의 TIM `0x11218` 은 **US와 100% 동일** (숫자 폰트라 안 건드림)
- CN 은 대신 **ASCII 폰트 TIM(`0x7F40`)을 68.2% 채움** (US 는 7.4%)
  → 영어를 포기하고 그 페이지를 한자로 도배
- CN 파일0 의 TIM 목록/크기는 US 와 완전히 같음 → **TIM 추가 없이** 기존 것만 재활용

### 인코딩 (팔레트 코드를 페이지 선택으로 겸용)
```
0xCF 01  → 팔레트 row7 + 페이지1 (tpage 0x000E, V = 144 + (idx>>4)*12)
0xCF 02  → 팔레트 row8 + 페이지2 (tpage 0x001F, V =  80 + (idx>>4)*12)
0xCF 03  → 팔레트 row9 + 페이지3 (tpage 0x001F, V =   0 + (idx>>4)*12)
0xCE <n> → 음절, idx = n - 0x90,  U = (idx & 0x0F) * 16
```

⚠️ **페이지 전환 코드는 2바이트를 먹는다.** 음절을 페이지에 마구 흩으면
인코딩이 원문보다 길어져 삽입이 안 된다.
→ **빈도순으로 배치해 전환을 최소화**할 것.
훅이 이미 `[0x8007B454]`(팔레트 변수)를 CLUT 계산에 읽으므로 **같은 값으로 페이지도 판정**.

### tpage 전환 (페이지2)
프리미티브에 tpage 필드가 없으므로 **DR_TPAGE(GP0 0xE1)** 를 큐에 삽입.
등록 함수 `0x8003D2C4` 는 **LIFO**(나중 등록이 먼저 실행)이므로:
```
등록 순서: 복원tpage(0x000E) → sprite → 설정tpage(0x001F)
실행 순서: 설정(0x001F) → sprite → 복원(0x000E)   ✓
```
- DR_TPAGE 프리미티브 (8B): `+0 tag=0x01000000`, `+4 cmd=0xE1000000|tpage`
- s0 전진 = 0x24 (sprite 20 + tpage 8 + tpage 8)
- 검증: 한 대사 안에서 `페이지2 → 페이지1 → 페이지2` 3회 전환 정상 출력

### 🔴 MIPS 함정 추가: LUI/오프셋 부호확장
`lw/lhu/lbu/sw/sh/sb` 의 offset 은 **부호있는 16비트**다.
하위16비트가 `0x8000` 이상이면 음수가 되므로 **LUI 값에 +1** 해야 한다.
```
0x8007B454 →  LUI 0x8007 ; LHU 0xB454  →  0x8006B454  ✗ (틀림!)
              LUI 0x8008 ; LHU 0xB454  →  0x8007B454  ✓
```
증상: 팔레트 변수를 엉뚱한 주소에서 읽어 CLUT/페이지 판정이 전부 틀어짐 → 한글이 안 보임.
→ `mips.py` 의 **`hi16()` / `lo16()`** 헬퍼를 쓸 것.

## 8-E. 🇯🇵 일본판 폰트 배치 해독 → 20열 방식 (400 음절)

### 일본판 폰트의 정체
- **위치: `PROT.DAT 0x007F34`** (파일0), VRAM (896,0) 256×256 tpage `0x000E`
- 비영 68.9%, 니블값 8종 — **한자 폰트 한 장뿐** (동적 로딩 아님)
- 미국판은 **같은 자리(`0x7F40`)가 7.4%** — 한자를 통째로 들어냄
  → 우리가 한글을 넣는 바로 그 자리다

⚠️ VRAM 뷰어에서 왼쪽에 보이는 "한자"는 **프레임버퍼를 4bpp 로 본 노이즈**다.
   VRAM (0,0)~(320,480) 은 프레임버퍼(더블버퍼). 폰트가 아니다.
   두 번의 VRAM 덤프 모두 폰트 페이지는 **(896,0) 하나뿐**.

### 일본판 렌더러 (`0x80037A00`) — U/V 계산
```
V = 0xC0 (192) 시작, U = 0
while idx >= 100: V += 64; idx -= 100
while idx >=  20: V += 12; idx -=  20
while idx >=   5: U += 64; idx -=   5
while idx >=   1: U += 12; idx -=   1
글리프 12 x 12
```
정리하면:
```
r = idx % 20
U = (r / 5) * 64 + (r % 5) * 12     ← 한 행에 20자
V = vbase + (idx / 20) * 12
```
- U/V 는 **1바이트**(sb) → 256 에서 wrap. V=192 시작해 한 바퀴 → **21행 × 20자 = 420자**
- MIPS 는 나눗셈이 느리므로 **반복 뺄셈**으로 구현

### 적용 결과: 320 → 400 음절
16열(`U = idx*16`)은 12px 글리프에 16px 를 배정해 **글자당 4px 낭비**.
20열로 바꾸면 U 가 0~240 에 촘촘히 들어간다.

| 페이지 | VRAM | tpage | 16열 | **20열** |
|---|---|---|---|---|
| 1 | ASCII TIM V144~255 | `0x000E` | 144 | **180** |
| 2 | 숫자 TIM V80~143 | `0x001F` | 80 | **100** |
| 3 | 숫자 TIM V0~79 | `0x001F` | 96 | **120** |
| | | **합계** | 320 | **400** |

**폰트 데이터도 VRAM 도 그대로. 계산식만 바꿔서 +80자.**
게임 검증 완료 (3페이지 전환 + 숫자 정상).

### 일본판 주요 주소 (참고)
| 항목 | 일본판 | 미국판 |
|---|---|---|
| EXE load | `0x80010000` (size 0x6D800) | `0x80010000` (size 0x6B800) |
| LoadImage | `0x8005B614` | `0x80059BD4` |
| 텍스트 버퍼 | `0x80075D94` | `0x800740EC` |
| 텍스트 전처리 | `0x80036D4C` 부근 | `0x80036514` |
| 글자 렌더 | `0x80037A00` 부근 | `0x80036888` |

일본판 텍스트는 **Shift-JIS 그대로**. (예: 記憶の像 = `8B4C 89AF 82CC 919C`)

### 🔴 실수 방지 가드 4종 (전부 빌드를 중단시킨다)

전부 **실제로 겪은 실수**를 코드로 막은 것이다.

| 가드 | 막는 것 | 겪은 증상 |
|---|---|---|
| `protected.assert_protected()` | 숫자폰트/TIM좌표/팔레트 파괴 | 메뉴 숫자가 전부 사라짐 |
| `check_macros()` | 매크로 삭제 | **게임이 멈춤** |
| `check_roundtrip()` | 인코딩 버그 전반 | 이름이 안 나오고 글자가 밀림 |

**`check_roundtrip()` 이 가장 강력하다**: `encode -> decode` 해서 원본과
일치하는지 본다. 매크로 바이트 수 오류, 인덱스 초과, 페이지 전환 누락을
한 번에 잡는다.
```
🔴 왕복 검증 실패 — 인코딩 버그!
   보냄: '봐라 {NAME}. 저것이'
   받음: '봐라 {NAME:2E} 저것이'   <- '.'(0x2E)를 파라미터로 먹음
```

### 🔴 매크로는 2바이트다 (덤프·인코딩 양쪽에서 겪은 버그)

```
원본: 1F "Look, " C1 00 ". That is" 00
                  └매크로 2바이트┘
```
- **덤프 버그**: 파라미터 `00` 을 문자열 종료로 오인 → `". That is"` 를 통째로 놓침
  → 런 수가 21,956 → **22,509** 로 늘어남 (553개를 놓치고 있었다)
- **인코딩 버그**: `0xC1` 만 1바이트로 넣음 → 전처리가 다음 글자(`.`)를
  파라미터로 먹어 **이름이 사라지고 텍스트가 밀림**

파라미터가 0 이 아닌 경우도 있다: `{NAME:63}` = `C1 63` ('c' 접미사)

### 🔴 페이지 전환이 번역 길이를 좌우한다

전환 코드가 **2바이트**를 먹으므로, 한 대사의 음절이 여러 페이지에 흩어지면
길이 초과가 난다. 짧은 대사일수록 치명적:
```
"메이" = 4B(글자) + 4B(전환 2회) = 8B  > 7B 한계  ✗
```
→ `assign_pages()` 가 **원문이 빠듯한 대사부터 처리하며, 그 대사의 음절을
   같은 페이지에 연속 배정**한다. 이것만으로 길이 초과가 52건 → 0건이 되었다.

### 🔴 팔레트: 기호(ASCII)와 한글의 색을 맞춰야 한다

ASCII 글리프는 **값14=외곽선, 값15=본체** 로 안티앨리어싱을 이룬다.
- 둘 다 흰색 → **글자가 뭉개짐**
- 원본 row6 색 그대로 → **기호가 한글보다 어두워 색이 안 맞음**
- ✅ **값15=흰색(본체), 값14=회색(외곽)** → 밝기는 한글에 맞추고 명암은 유지

### 텍스트 재배치는 불가능 (in-place 만 가능)
텍스트가 스크립트 **코드 안에 인라인**으로 박혀 있어(런 사이 간격 평균 1~3B),
길이를 바꾸면 뒤쪽 점프/분기 주소가 전부 깨진다.
→ **번역은 원문 길이 이하로** 압축해야 한다. (스크립트 엔진 전체 해독은 대공사)

### 🔴 PACK 재사용 시 0 패딩 버그
원본 압축 데이터를 재사용할 때 **마지막 엔트리**를 `raw[addr:]` 로 자르면

→ 증상: 번역이 전부 되돌려짐 (195런 → 37런)
→ `pack_end = len(raw.rstrip(b"\x00"))` 으로 실제 끝을 구할 것.

### ⚠️ 슬롯이 꽉 찬 맵 → ✅ 8-O PROT 재배치로 해결
일부 맵은 **원본 PACK 이 슬롯을 꽉 채웠다** (여유 0~11B).
한글은 영어만큼 압축되지 않아 재압축 결과가 슬롯을 넘는다.
```
map350: 슬롯 237,568B / 원본 237,568B (여유 0B)  -> 26런 삽입 실패

## 8-I. ✅ 동적 폰트 로딩 — **게임에서 검증 완료**

**맵 로드 시점에 VRAM 폰트를 교체할 수 있다.** 이걸로 400자 제한이 사라진다.

`tools/hook_dynload6.py` — 게임에서 확인한 최종 형태:
```
LoadImage 진입
  → 재진입 락 (무한 재귀 방지)
  → 원본 LoadImage 완료
  → 우리 폰트 전송            ★ 매 호출마다
  → 락 해제
```

### 🔴 매 호출마다 덮어써야 한다
처음에는 '첫 LoadImage 에서 한 번만' 전송했는데 **대사가 멀쩡했다**.
게임이 그 **뒤에** 폰트 TIM 을 로드하면서 우리 데이터를 지웠기 때문이다.
→ 락은 재진입 방지용으로만 쓰고, **매 호출마다** 전송해야 게임을 이긴다.

검증: 폰트 첫 줄 20자(봐 라 저 것 이 메 조 심 해 들 어 갈 수 없 다 잠 겨 있 무 엇)를
노이즈로 덮었더니 **정확히 그 글자만** 깨졌고, **전투 모델은 멀쩡했다**.

---



### ✅ 밝혀낸 것
**게임이 LoadImage 를 부르는 시점(맵/세이브 로드)에 VRAM 을 건드려도 화면은 안 깨진다.**
렌더링 **도중** LoadImage 가 화면을 파괴하는 것과 대조적이다.
→ **맵별 동적 폰트가 가능하다.**

`LoadImage(0x80059BD4)` 의 호출자는 **`0x80085E18`** — EXE(~0x8007B800) **범위 밖**.
즉 **오버레이 코드**다. 그래서 정적 분석(jal 검색)으로는 못 찾았다.
에뮬레이터 브레이크포인트가 결정적이었다.

### 🔴 함수 진입점을 훅할 때의 딜레이 슬롯 (PoC 를 세 번 헛돌게 한 버그)

```
원본:  80059BD4: addiu sp, sp, -0x50     <- 이걸 `j 훅` 으로 덮음
       80059BD8: sw    s1, 0x34(sp)      <- 🔴 딜레이 슬롯! 그대로 실행된다
```

MIPS 는 점프의 **다음 명령을 반드시 실행**한다. 그래서 `sp` 를 아직 내리지
않았는데 `sw` 가 실행되어 **호출자의 스택 프레임이 오염**됐다.

증상이 고약했다 — **대사는 멀쩡한데 전투 캐릭터/몬스터 모델만 조각남**.
전송 위치를 바꿔도(V216 → V144), 순서를 바꿔도(원본 먼저 → 나중),
LoadImage 대신 GP0 직접 전송을 써도 **똑같이 깨졌다**. 원인이 딴 데 있었으니까.

**올바른 처리**
```
1. 진입점 두 번째 명령을 nop 으로 덮는다
2. 훅이 원본 1·2번째 명령을 모두 수행한 뒤 세 번째로 복귀한다

   L_orig:
     addiu sp, sp, -0x50      # 원본 1
     j     0x80059BDC         # 원본 3 으로
     sw    s1, 0x34(sp)       # 원본 2 — j 의 딜레이 슬롯 (sp 조정 뒤라 안전)
```

`asm.py` 의 **`check_hook_site()`** 가 이걸 강제한다.
(대사 훅은 딜레이 슬롯이 `addiu v0, zero, 0x20` 이라 무해해서 우연히 잘 돌았다)

### 🔴 p4 (VRAM 896, V216~252) 는 전투 중 캐릭터 텍스처 자리
PoC1 에서 그 자리를 덮었더니 **전투 모델이 조각났다**.
전투 VRAM 덤프로 확인: p1(V144~216), p2, p3 는 전투 중에도 **한글이 보존**되지만
p4 는 캐릭터 텍스처가 덮어쓴다. → **p4 는 제거해야 한다.**

### ✅ ASCII 영역(896, V0~143)은 전투 중에도 안전
전투 VRAM 에서 값 2종 · ASCII 글리프 그대로. → **회수하면 +240칸**

### 새 배치 안 (580칸)
| 페이지 | 위치 | 칸 |
|---|---|---|
| ASCII 회수 | (896, V0~143) | **240** |
| p1 | (896, V144~216) | 120 |
| p2 | 숫자TIM V80~143 | 100 |
| p3 | 숫자TIM V0~79 | 120 |
| ~~p4~~ | ~~V216~252~~ | ~~60~~ 🔴제거 |
| | **합계** | **580** |

단 ASCII 를 잃으므로 **메뉴/아이템/전투 텍스트도 한글화**해야 한다.

## 8-J. ❌ 맵별 폰트 저장소 — DMY.DAT (폐기: 8-S2 정적 폰트로 대체)

동적 로딩은 검증됐다(8-I). 이제 **맵별 폰트 데이터를 어디에 둘지** 정해야 한다.

### PROT 재배치는 부족
맵마다 폰트를 넣으려면 PROT 를 재배치해야 하는데, 섹터(2048B) 정렬 낭비 때문에
가변 크기로 최적화해도 **약 295KB 초과**한다 (PROT.DAT 크기는 디스크상 고정).
```
맵별 가변 폰트 합계: 334KB
재배치 후 PROT: 121,548,800B > 121,253,888B  → 295KB 초과
```

### ★ DMY.DAT (36MB) 가 답
```
DMY.DAT  LBA 180228~198283  36,975,078B  — 게임이 읽지 않는 더미 파일
```
82맵 × 8KB = 656KB 로 충분히 들어간다. 남는 공간이 압도적.

**필요한 것**: 맵 로드 시 DMY.DAT 의 해당 오프셋에서 폰트를 읽는 CD 코드.
- 게임의 CD 읽기는 BIOS 콜(0xA0/0xB0/0xC0) 기반 (하드웨어 레지스터 직접 접근 없음)
- 게임에 이미 PROT.DAT 를 읽는 로더가 있으니 그 경로를 재사용하는 게 안전
- ~~[TODO] 로더 함수 시그니처 파악 → 폰트 읽기 훅 작성~~ (폐기: 8-S2 정적 폰트로 대체)

### 당장 가능한 절충안 (CD 코드 없이)
PROT 여유(513KB)에 **자주 나오는 맵 위주로** 폰트를 넣고,
용량 큰 맵은 공용 400자로 커버 → 대부분의 대사를 한글화할 수 있다.
전체 완성은 DMY.DAT CD 로딩이 필요.

## 8-K. ❌ CD 동적 로딩 — DMY.DAT 읽기 성공 (폐기: 8-S2 정적 폰트로 대체)

DMY.DAT 에서 폰트를 읽어 오는 것 자체는 **작동한다**. CD 상태창으로 확인했다:
`Current File: DMY.DAT (2,891,776 읽음)`.

### ✅ 확정된 CD 함수 (libcd)
```
CdRead    0x8005E4D4  (a0=섹터수, a1=LBA, a2=버퍼)  v0=1 성공, 동기
CdReadSync 0x8005EA84  (a0=mode, a1=buf)  v0=0 이면 idle
CdIntToPos 0x8005C328  (LBA -> BCD min:sec:frame)
```

### DMY.DAT 폰트 레이아웃 (`tools/dmy_font.py`)
```
DMY.DAT  LBA 180228, 36MB — 게임이 안 읽는 더미
  섹터 0        : "KFNT" 매직 + 맵 목록
  섹터 1~       : 맵별 폰트 (각 24섹터 = 660자 raw 4bpp)
82맵 x 24섹터 = 4MB  (DMY 36MB 의 11%)
```

### 🔴 LoadImage 는 CD 훅 지점으로 부적합
LoadImage 는 로고/타이틀/**데모 영상**/게임 어디서나 불린다.
- 부팅 로고: 카운터(BOOT_SKIP)로 넘김 → 통과
- **타이틀 데모 영상(MOV/MV1.STR)**: XA 스트리밍 중 CdRead 하면
  CD 헤드가 DMY 로 이동 → **영상이 끊기고 멈춘다**

게임플레이 대사는 완벽하다(봐라 Vahn). 문제는 오직 영상 중 CD 가로채기.

### 🔴 데드락 → 화면깨짐 → 영상끊김: 3연속 타이밍 버그
1. **데드락**: 부팅 초기 CdRead → 게임 CD 와 충돌 (0.00 FPS)
   → 부팅 카운터로 해결
2. **로고 깨짐**: `L_xfer_skip` 라벨이 전송 앞에 있어, CD 안 읽어도
   쓰레기 버퍼를 VRAM 에 전송 → 라벨을 전송 뒤로 옮겨 해결
3. **영상 끊김**: LoadImage 가 영상 중에도 불림 → 미해결

> ※ 이 CD 동적 로딩 접근은 **폐기**됐다. 8-S2 에서 일본어 한자 폰트 TIM
>   2페이지(1,100칸)를 찾아 폰트를 정적으로 확장하는 쪽이 훨씬 간단하고
>   안정적이라, CD 동적 로딩은 더 이상 쓰지 않는다. (영상 끊김도 자연히 무의미)

`asm.py` 의 **`check_cd_hook_safety()`** 가 부팅보호+실패스킵을 강제한다 (가드 6).

### 🔜 남은 실측: 영상/게임플레이 구분
LoadImage 는 부적합하므로, "게임플레이 중"을 나타내는 신호가 필요하다.
- [실측 A] 타이틀 영상 재생 중 vs 후, 어떤 전역이 바뀌나
- [실측 B] 맵 번호 전역 (맵 바뀔 때만 폰트 교체)
- 그 신호가 확실할 때만 CdRead

### 대안 (CD 없이): 공용 폰트 EXE 상주
영상 문제를 피하려면, 공용 400~580자를 EXE 에 상주(hook_dynload6, 검증됨)시키고

## 8-M. 🔬 ASCII 회수 실험 — 회수는 '마지막 단계'에

그래서 ASCII TIM 픽셀을 노이즈로 바꿔 **뭐가 깨지는지** 실험했다.

### ASCII TIM 위치
```
PROT 파일0  TIM 헤더 0x006740,  픽셀 0x006780
            VRAM (896,0)  64x256 halfword = 256x256 px
            V0~143   = ASCII (12행)      ← 회수 대상
            V144~215 = 한글 p1
            V216~252 = p4 (전투 위험)
```

### 실험 결과
| 대상 | 결과 |
|---|---|
| 숫자 (HP 1555, LV 27, G 602) | ✅ 멀쩡 — 별도 TIM(960,256) |
| 라벨 (HP/MP/LV/G/TIME/AP) | ✅ 멀쩡 — 별도 TIM |
| 메뉴·아이템·기술명 | ❌ 깨짐 |
| 캐릭터 이름 (Vahn) | ❌ 깨짐 |
| 대사 중 번역 안 된 영어 | ❌ 깨짐 |

**ASCII 폰트는 '모든 영어 텍스트'에 쓰인다.**

### 빈 칸은 30개뿐
12x12 셀 단위로 보면 21열 x 12행 = 252칸 중 **222칸이 사용 중**,
빈 칸은 **30개**. 즉 공짜로 얻을 수 있는 건 400 → 430칸뿐이다.

### 🔴 결론: ASCII 회수는 전체 번역이 끝난 뒤에
ASCII 240칸을 다 회수하려면 **모든 영어를 없애야** 한다:
1. 대사 17,313개 전부 번역 (하나라도 남으면 그 자리가 노이즈)
2. 메뉴/아이템/기술 1,109개 번역

## 8-N. ✅ 번역 시작 — 중복 전파 + LZSS 개선으로 슬롯 문제 해결

### 중복 대사가 아주 많다
같은 원문이 반복된다. 하나 번역하면 여러 곳이 채워진다.
```
'{NAME:63} now has'            253회
'The treasure chest is empty!' 191회
'in the treasure chest!'       188회
'Exit'                         152회
```
→ GUI 에 **"같은 원문에 자동 적용"** 토글과 **"★ 빈출순"** 뷰를 넣었다.
   빈출 대사 20개만 번역했더니 **1,364런(6.1%)** 이 채워졌다.

### 🔴 한글은 압축이 안 돼 PACK 이 슬롯을 넘는다
맵 PACK 슬롯은 원본에서 이미 꽉 차 있다 (**맵 여유 합계 3,313B**).
한글은 영어만큼 LZSS 압축이 되지 않아 PACK 이 커지고, 슬롯을 넘으면
빌드가 번역을 되돌린다 → **57개 맵에서 807런 손실**.

### ✅ 해결: lazy matching LZSS (`lzss_compress_opt`)
게임 디코더는 **표준 LZSS** 라 어떻게 인코딩했든 똑같이 풀린다.
원본 압축기는 greedy(항상 가장 긴 매치)인데, lazy 로 바꾸면:
```
현재 위치의 매치 길이 L1 을 찾았어도,
다음 위치에서 더 긴 매치(L2 > L1)가 가능하면
현재 바이트를 리터럴로 내보내고 한 칸 전진한다.
```
→ **약 1.2% 작아진다** (map6: 24,891 → 24,602B). 왕복 검증 통과.

결과:
```
이전: 슬롯 부족 57개 맵, 807런 손실
지금: 슬롯 부족  5개 맵,  37런 손실   ← 95% 해결
```
🔴 C 구현(lzss_fast)은 원본과 같은 greedy 라 더 크다. **파이썬 lazy 를 쓴다.**
   (캐시가 있어 재빌드는 빠르다)

### 🔴 짧은 UI 텍스트는 페이지 전환 2B 때문에 안 들어간다
한글 1자 = 2B 지만, **페이지가 바뀌면 `0xCF <page>` 2B 가 더 붙는다**.
```
'Exit'   (4B) → '나감' 6B  ✗   → '끝' 4B  ✓
'=Next=' (6B) → '=다음=' 8B ✗   → 영어로 남김
```
짧은 텍스트는 자주 쓰는 페이지의 음절로 맞추거나, 영어로 둔다.

### 남은 슬롯 문제: PROT 재배치
5개 맵(map80/238/520/554/683)이 아직 넘친다. 총 초과는 작다.
`tools/prot_realloc.py` — 파일을 섹터 정렬로 다시 배치해 맵에 슬롯을 준다.
빈틈 없이 재배치하면 **106KB** 가 생긴다. → ✅ 8-O 에서 `build_kr.py` 에 통합 완료.

## 8-O. ✅ PROT 재배치 — 슬롯 부족 완전 해결

### 문제
맵 PACK 슬롯이 원본에서 이미 꽉 차 있다 (**맵 여유 합계 3,313B**).
한글은 LZSS 압축이 덜 돼 PACK 이 커지고, 넘치면 빌드가 번역을 되돌린다.

### 초과량은 아주 작다
```
슬롯 넘는 맵 56개, 총 초과 1,518B   (맵당 평균 27B)
```
하지만 **섹터(2048B) 정렬** 때문에 각 맵이 1섹터씩 더 받아야 한다.
→ 56 x 2048 = 114,688B 필요

### PROT 재배치 (`tools/prot_realloc.py`)
파일 내용은 그대로 두고 **오프셋만 다시 잡는다**. 원본이 낭비하던 공간을 회수:
```
빈틈없이 재배치      → 106,496B 회수
빈 파일 4개 0섹터로  →   8,192B 추가   (원본은 빈 파일에도 1섹터씩 줬다)
                      ─────────────
                       114,688B  ← 필요량과 정확히 일치
```

**PROT 헤더 구조** (중요):
```
u32[0]   = 0 (미사용)
u32[1]   = 파일 수 (1235)
u32[2..] = 각 파일의 섹터 오프셋 (offset << 11)
첫 파일은 오프셋 6144 (헤더 3섹터) 부터
```
빈 파일은 오프셋을 다음 파일과 같게 두면 **크기 0** 으로 파싱된다 (파서 정상 동작).

### 결과
```
이전: 슬롯 부족으로 76런 손실
지금: 1,681런 전부 삽입 (손실 0)  ✓
```
`tools/build_kr.py` — 2패스 빌드:
1. 번역 넣었을 때 각 맵이 필요한 슬롯 측정
2. PROT 재배치 → 슬롯 확보 → 번역 삽입
3. 모든 파일 내용 보존 검증 (`PR.verify`)

⚠️ 지금 여유가 **0B** 다. 번역이 늘면 다시 부족해진다.
그때는 LZSS 최적 파싱(DP)으로 더 줄이거나, 긴 번역을 다듬어야 한다.

## 8-T. ✅ PAL_VAR 은 "색"과 "페이지"를 겸한다 — 페이지 8~11 뒤 영어가 초록 (해결)

### 증상
한글은 흰색으로 잘 나오는데, **뒤따르는 영어/기호가 초록**으로 물든다.
페이지 8~11(한자 TIM2, tpage 0x16)을 쓰는 대사에서만 나타난다.
p8~p11 만 격리해 빌드하니 확실히 재현됐다 (LEGAIA_TEST_P8=1).

### 원인
PAL_VAR (0x8007B454) 은 원본 대사 렌더러가 **글자 색**으로 쓰는 변수다.
```
원본 clut 공식:   clut = [PAL_VAR] + 0x7F86      (0x80036B80)
0xCF <n> 제어코드: PAL_VAR = n  으로 색을 바꾼다   (0x8003695C)
```
우리는 **페이지 전환에 같은 0xCF <page>** 를 쓴다.
→ 한글 페이지 8~11 을 그리면 PAL_VAR = 8~11 이 된다.
→ 그 뒤 영어(원본 렌더러)의 clut = 0x7F86 + 8~11 = 0x7F8E~0x7F91
  = 그 CLUT row 는 흰색이 아니다 → 영어/기호가 초록.

한글 sprite 자체는 우리가 clut 을 직접 지정(0x7F87)하므로 흰색으로 나온다.
문제는 **그 다음 원본이 그리는 영어**다.

### 왜 p1~p7(KR24 배포용)은 정상이었나
page 1~4 → clut 0x7F87~0x7F8A → CLUT row 7~10.
translate 가 row 7~10 을 흰색으로 칠했고, 대사 배치상 page 5~7 뒤에
영어가 드물어 안 드러났다. p8~p11 을 격리하니 드러났다.

### 관련 주소
```
0x8003695C  sw v1, PAL_VAR      0xCF <n> 처리: PAL_VAR = n (색/페이지)
0x80036B74  lhu v1, PAL_VAR     clut 계산 시 읽음
0x80036B80  addiu v1, v1, 0x7F86  clut = PAL_VAR + 0x7F86
0x8007B454  PAL_VAR
```

### ✅ 해결 (완료) — 방향 A 채택: 0xCF 를 훅으로 가로채 PAGE_VAR 로 분리

페이지 저장을 PAL_VAR 이 아닌 **별도 변수 PAGE_VAR** 로 옮기고, 0xCF 처리를
CF_HOOK 으로 가로챈다. PAL_VAR(색)은 오염되지 않게 지킨다.

**1) 인코딩 (`translate.py` encode/decode)**
페이지 전환을 `0xCF <0xF0|page>` 로 인코딩한다.
원본은 0xF0~0xFB 대역을 색으로 쓰지 않으므로 이 대역이 안전하다.
```
page p  →  0xCF <0xF0|p>     (p=1~11 → 0xF1~0xFB)
```

**2) CF_HOOK (`hook7.py`)**
```
0x80036958 'j LOOP' → 'j CF_HOOK'
  (딜레이슬롯 0x8003695C 의 sw PAL_VAR=n 은 그대로 실행된 뒤 진입)

CF_HOOK:
  n >= 0xF0 : 페이지 전환. PAGE_VAR = n & 0x0F, PAL_VAR = 1 (색 복원)
  n <  0xF0 : 원본 색코드. PAL_VAR = n 그대로 (색 연출 보존)
```
→ 우리 페이지 값은 PAL_VAR 을 오염시키지 않고, 원본의 색 연출(n<0xF0)도 산다.

**3) 한글 렌더**
페이지를 PAL_VAR 대신 **PAGE_VAR 에서 읽는다** (`t5 = [PAGE_VAR]`).

```

## 8-U. 🔴 일본판 전환 결정 — 미국판 슬롯의 근본 한계

### 결론 먼저
미국판은 대사를 아무리 줄여도 **한글의 절반밖에 못 담는다.**
"대사를 줄여야만 하는 한글화"는 제대로 된 한글화가 아니다.
**일본판으로 전환하면 한글이 1:1 로 들어간다.** 그래서 전환한다.

### 근본 원인 — 슬롯이 "문자 크기" 기준이다
대사 슬롯 = 원문이 스크립트에서 차지하던 **바이트 수**. 그런데:

| 판본 | 원문 문자 크기 | 한글(2B/음절)과 궁합 |
|------|--------------|---------------------|
| 미국판 | 영어 **1바이트/문자** | 한글 2B → **밀도 2배 초과** ✗ |
| 일본판 | 일본어 **2바이트/문자** | 한글 2B → **1:1 거의 일치** ✓ |

미국판 영어 슬롯은 1바이트/문자 기준으로 잡혀서, 2바이트/음절인 한글이
근본적으로 안 맞는다. 페이지 전환(2B)까지 겹쳐 더 나빠진다.

### 실측 증거 — 같은 대사, 두 판 비교
일본판 PROT.DAT (디스크 LBA 246, ISO9660 디렉토리로 확인) 를 추출해
같은 맵(map6)의 같은 스크립트를 비교:

```
미국판 s1 (516B): "The house is locked and you can't get inside." (영어 46자)
일본판 s1 (508B): "鍵が　かかっている。家の中には　入れない。"      (일본어 20자)
```

- 같은 뜻인데 **바이트 수가 거의 같다** (516 ≈ 508).
- 영어는 46자×1B, 일본어는 20자×2B — 문장이 짧아진 만큼 문자가 커져 상쇄.
- map6 전체 스크립트 128개 중 **88% 가 일본판 ≥ 미국판** (평균 345B vs 354B).

핵심: 일본어 N문자 자리 = **한글 N음절** 이 그대로 들어간다.
일본판은 애초에 "문자당 2바이트" 공간이라 한글과 밀도가 정확히 같다.

### 미국판에서 시도했던 완화책 (한계 확인)
1. **LZSS lazy matching** (§8-N): 압축률 개선 — 부분적
2. **PROT 재배치** (§8-O): 빈 슬롯 회수로 106KB 확보 — 전역엔 부족
3. **INIT_HOOK** (아래): 첫 페이지 전환 생략 — 초과 1,283→958런
4. **번역 길이 다듬기**: 문맥 훼손 위험, 근본 해결 아님

이 모두를 적용해도 미국판은 대사의 **약 78% 만** 삽입.
나머지는 영어로 남는다 → 한글화로서 불완전.

### 일본판 전환 비용 (정직하게)
사실상 렌더링 계층은 처음부터 재구축:
1. 모든 훅/주소가 미국판 EXE 기준 (렌더러 `0x80036888`, hook7, CF/INIT_HOOK,
   폰트 TIM 위치, PROT 재배치) → 일본판 EXE(`SCPS_100.59`)에서 재분석
2. 일본판은 제어코드·인코딩이 다름 (Shift-JIS 계열 2바이트, 제어코드 체계 상이)
3. 번역 4,348런은 미국판 영어 런 기준 → 일본판 스크립트에 재매핑
4. 폰트 렌더러도 2바이트 문자 처리 방식이 달라 재검증

### 보존되는 자산 (재작업 아님)
- **번역 텍스트 자체** (한글 대사) — 스크립트 재매핑만 하면 재사용

- **듀얼 레이어 지식** (§8-V) — 일본판은 이걸 실제로 쓰므로 오히려 필수 지식

## 8-V. ★★ 듀얼 폰트 패킹 — 한 텍스처에 폰트 2장 (폰트 2배)

### 핵심
한 4bpp 픽셀(값 0~15) 안에 **3단계 계조 글리프 2장**을 겹쳐 저장하는 기법.
하위 2비트 = 레이어 A, 상위 2비트 = 레이어 B. **CLUT만 바꿔** 원하는 레이어를 그린다.
→ 폰트 한 장 넣을 VRAM 에 **두 장**을 담는다.

### 발견 경위
PCSX-Redux VRAM 뷰어에서 한자 밑에 히라가나/카타카나가 희미하게 깔린 게
보였다. 뷰어가 4비트 인덱스를 밝기 램프(idx×17)로 그려서 두 레이어가
겹쳐 보인 것. 니블 값 히스토그램을 뜯어보니 규칙이 드러났다.

### 증거 1 — 니블 값이 규칙적으로 제한됨
폰트가 쓰는 픽셀 값: **{0, 2, 3, 8, 10, 11, 12, 14, 15}** 뿐.
`{1, 4, 5, 6, 7, 9, 13}` 은 아예 안 씀. 비트로 쪼개면:
```
값 = (상위 2비트) × 4 + (하위 2비트)
하위 2비트 ∈ {0, 2, 3}  ← 3단계 계조 (레이어 A)
상위 2비트 ∈ {0, 2, 3}  ← 3단계 계조 (레이어 B)
```
금지값(1,4,5…)은 "한 레이어가 값 1(=단계 1)을 쓰는 경우"인데,
폰트가 3단계 {0,2,3}만 쓰므로 그 조합이 안 나온다.

### 증거 2 — 실제로 두 글자가 분리됨
- 하위 2비트 레이어 A: 기호·ASCII·히라가나·카타카나 + 한자 세트 A
- 상위 2비트 레이어 B: 한자 세트 B

### 증거 3 — CLUT 이 레이어를 골라 표시
폰트의 16색 팔레트는 색이 인덱스의 **하위 2비트에만** 의존
(idx&3: 0=투명, 3=회색). 상위 2비트는 무시 → 레이어 A만 화면에 나온다.
한자 레이어(B)를 그릴 땐 색이 **(idx>>2)&3** 을 따르는 다른 팔레트로 교체.
CLUT 은 32바이트라 교체 비용이 사실상 공짜.

### 왜 이렇게 만드나 — VRAM 절약
PS1 VRAM 은 1MB. 폰트는 3~4단계 계조면 충분한데 4bpp 는 16단계라 비트가 남는다.
그래서 3단계 폰트 2장을 한 4bpp 텍스처의 하위/상위 비트에 채우고,
CLUT 만 바꿔 원하는 장을 그린다. **미국판은 안 쓴다** (영어 26자라 불필요).
**일본판·중국 패치가 쓴다** (한자 수천 자라 VRAM 을 아껴야 함).

### 우리(미국판 작업)에의 함의
- 우리는 잉크로 **값 3(하위 2비트=11)만** 썼다 → 이미 레이어 A 만 사용.
  상위 레이어(값 8,12 등)는 통째로 비어 있었다.
- 즉 CLUT_B((idx>>2)&3, 값 12=흰색)를 추가하면 **폰트가 거의 2배**:
  한자 TIM 840칸 → 1,680칸, 전체 1,100 → 최대 ~1,940칸.
- **한글은 값 0/값 3만 써서 금지값을 안 만든다** → 규칙 이미 준수.

### 🔴 검증됨: 듀얼 레이어는 대사 길이에 영향 0
- 음절 인코딩은 레이어 무관하게 `0xCE <idx>` = **2바이트** 그대로.
- 음절이 폰트 칸(현재 1,100)을 넘기 전엔 레이어 B 를 안 쓰므로 배치 불변
  (실측: 전환 수 7,031 → 7,031 동일).
- 나중에 레이어 B 를 써도 "같은 VRAM 2배"일 뿐, 전환은 음절 분포가 결정.
- **결론: 폰트 공간을 2배로 늘리되 길이는 중립.** (전환 최적화는 §INIT_HOOK 로 별개)

### 일본판에서의 위치
일본판은 이 기법을 **실제로 쓴다.** 그러므로 일본판 폰트를 한글로 교체할 때
듀얼 레이어 구조(하위/상위 레이어, CLUT_A/CLUT_B)를 반드시 이해하고
다뤄야 한다. 미국판에서 얻은 이 지식이 일본판 작업의 핵심 자산이다.

## 8-W. ✅ INIT_HOOK — 대사 시작 시 PAGE_VAR=1 (첫 전환 생략)

### 문제
페이지 전환 코드(`0xCF 0xF0|page`, 2B)가 전체 인코딩 바이트의 **26%**(20,862B).
그중 상당수가 "대사 첫 글자의 전환"인데, 대사의 **84%가 p1 으로 시작**한다.

### 해결
- encode 기본 `page=1` — 첫 음절이 p1 이면 전환 코드 생략.
- 렌더러 진입부 `0x800368F4 'move s4,zero'`(대사당 1회) 를 INIT_HOOK 으로
  돌려 대사 시작 시 **PAGE_VAR = 1** 초기화. (딜레이슬롯 `move s2,fp` 보존)
- decode 도 `page=1` 로 맞춰 왕복 유지.

### 효과
```
삽입 3,065 → 3,390런,  길이초과 1,283 → 958런 (325런 개선)
```
디스어셈블로 설치 확인:
```
0x800368F4: j INIT_HOOK
0x800368F8: move s2, fp        (딜레이슬롯, 그대로 실행)
INIT_HOOK: move s4,zero; PAGE_VAR=1; j 0x800368FC
```
(이 최적화는 미국판 기준. 일본판 전환 후 해당 렌더러에서 재적용.)
