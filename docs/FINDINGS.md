# 🔴 절대 건드리면 안 되는 것 (PROT.DAT)

**전부 실제로 건드렸다가 게임이 깨진 구간이다.**
빌더는 `tools/protected.py` 의 `assert_protected()` 로 반드시 검증할 것.

## 1. 숫자 폰트 — `0x015C38 ~ 0x019438` (14,336 B)

TIM `0x11218` (VRAM 960,256) 의 **V144~255** 영역.
**메뉴의 HP / MP / LV / G / TIME 숫자를 그리는 폰트다.**
지우면 숫자가 전부 사라진다.

```
TIM 0x011218 (256x256 4bpp, VRAM 960,256)
  CLUT  0x01122C (512B)      ← 대사 팔레트. 잉크값(3) 한 칸만 수정 허용
  IMG   0x01142C
    ix @ 0x011430 = 960      ★변경 금지 (옮기면 숫자 UV 가 전부 깨짐)
    iy @ 0x011432 = 256      ★변경 금지
  픽셀  0x011438 ~ 0x019438  (32,768B, 128B/행)

  픽셀 오프셋 = 0x011438 + V * 128

  V   0~ 79 : 0x011438 ~ 0x013C38   개발 메모   → ✅ 사용 가능 (페이지3)
  V  80~143 : 0x013C38 ~ 0x015C38   빈 영역     → ✅ 사용 가능 (페이지2)
  V 144~255 : 0x015C38 ~ 0x019438   🔴 숫자 폰트 → ❌ 절대 금지
```

⚠️ 이 TIM 은 겉보기에 "개발용 디버그 폰트"처럼 보인다
(일본어 메모 "ここは常駐エフェクトが入る予定 Pochi" 가 섞여 있음).
**속으면 안 된다. V144 아래는 게임이 실제로 쓰는 폰트다.**
CN 패치도 이 TIM 은 US 와 100% 동일하게 두었다.

## 2. 팔레트 — 잉크값(3) 한 칸만 수정

CLUT `0x01122C` (16색 × 16행).
row7,8,9 는 원본에 색이 꽉 차 있고 **게임 UI 가 쓴다.**
통째로 덮으면 메뉴가 깨진다. → **값3(한글 잉크) 한 칸만** 바꿀 것.

또한 **다른 팔레트의 값3 을 투명화하지 말 것.**
UI 폰트 TIM(`0x18E0`)이 값3 을 2072px 쓰고 있어 숫자가 사라진다.
(투명화는 애초에 불필요 — 한글/ASCII 는 V 영역이 분리되어 있다)

## 3. 디스크 구조

`XA` 는 빈 디렉토리가 **아니다** — 오디오 34개(~160MB) 를 담고 있다.
`MOV` 도 동영상 6개. **디스크 실제 빈 공간은 2섹터뿐.**
→ PROT.DAT 확장은 불가능. TIM 추가도 불가능.

## 2. PROT.DAT → PACK → LZSS → Script

```
PROT: u32[0]=0, u32[1]=파일수(1235), u32[2..]=섹터오프셋 (offset<<11)
PACK: u32 fileCount, u32 totalDecLength, per entry: u24 decLength, u8 fileId, u32 compAddress
Script: 0x22 u16 count0/1/2, 0x28 u24 footer, 0x2B.. u24 포인터테이블, base=0x2B+count*3
```
맵 파일 **fileId 3 = 텍스트/스크립트**

**LZSS**: 링버퍼 4096, 시작 4078(0xFEE), 플래그 LSB-first,
bit=1 리터럴 / bit=0 u16LE `len=((v>>8)&0xF)+3, pos=((v&0xF000)>>4)|(v&0xFF)`,
**종료조건 = 해제후 크기**

## 3. 텍스트 (US = 평문 ASCII)

```
<1F> 텍스트런 시작 ... <00> 종료
예: <1F>The Memory Statue will<00><1F>remember things for you...<00>
```

| 코드 | 의미 |
|------|------|
| `0x00` | 문자열 종료 |
| `0x7C` | 제어 (1바이트) |
| `0xCE <n>` | 테이블 조회 삽입 (`0x80074050 + n*4`) |
| `0xCF <n>` | **팔레트 전환** → CLUT = n + 0x7F86 |
| 그 외 | **전부 글리프로 렌더** |

**매크로**: `0xC1`=플레이어이름(2482회), `0xC2`=아이템명(457), `0xAC`(503), `0xC7`(454)

## 4. 🔑 대사 렌더러 = `0x80036888`

**찾은 방법**: PCSX-Redux에서 텍스트버퍼 `0x800740EC` 에 Read 브레이크포인트.

```
텍스트 버퍼 = 0x800740EC   (하드코딩)

문자 c 렌더:
    U = (c & 0x0F) * 16
    V = (c & 0xF0) - 0x20
    글리프 14 x 15 px, GP0 0x64 스프라이트

CLUT = [0x8007B454](u16) + 0x7F86     ← 팔레트가 "변수"
    0xCF <n>  →  0x8007B454 = n

폭 = [0x80073F1C + c]                  ← 가변폭 테이블
    EXE 파일오프셋 0x6471C, 디스크 LBA 224
X += 폭 + 1
```

**배제된 후보**: `0x8001A374`(8x8 HUD폰트), `0x8001A078`(printf류),
`0x8001ABC8`/`0x8001C7A0`(숫자), `0x8001BE80`(3D정점)

## 5. 폰트

- PROT.DAT 파일0 내 TIM: 헤더 `0x7F40`, **픽셀 `0x7F80`**, 256×256 4bpp
- VRAM (896,0), tpage 0x0E
- 타일에디터로 열 때: 파일0-상대 오프셋 `0x6780`
- ASCII 배치: 16열 격자, U/V는 위 공식대로

### 니블 값 (핵심)
- **US 원본은 값 {0,1,2,14,15} 만 사용** → 잉크 = **14, 15**
- **224개 셀 전부에서 값 1/2/3 이 완전히 비어 있음** (검증됨)
- CN 패치는 16값 전부 사용 (듀얼레이어)

### 팔레트
- TIM `0x11218` 의 CLUT = 16색 × **16팔레트**, VRAM (0,510)
- **PROT 오프셋 `0x1122C`**
- CLUT id = `arg + 0x7F86`, 팔레트 row = `arg + 6`
- 대사가 쓰는 arg: 0(기본), 4,5,6,7,9 (색 강조)
- **arg 1,2,3,8 (row 7,8,9,14) 미사용** → 한글용으로 재정의 가능

## 6. ✅ (A)방식 한글 출력 — 성공

**ASM 훅 불필요.** 폰트 + 팔레트 + 텍스트 + 폭테이블만 수정.

1. **한글을 니블 값 1/2 로** 폰트에 그림 (ASCII는 14/15 → 충돌 없음)
2. **팔레트 row 7** (`0xCF 01`) 을 한글 전용으로:
   값1=흰색, 값2=회색, **값14/15=투명**
3. **기존 사용 팔레트**(row 6,10,11,12,13,15)의 값1/2/3 → **투명화**
   (안 하면 한글이 영어 대사에도 나타남)
4. 텍스트: `0xCF 01` + 한글코드 + `0xCF 00`
   **길이를 원문과 동일하게** → 포인터 재계산 불필요
5. **폭 테이블 패치 필수**: 한글 코드 폭 → 13px
   ⚠️ 안 하면 글자가 겹침 (v1 실패 원인)

빌드: `tools/build_us_kr2.py` → `tools/write_us.py`

## 7. ⚠️ 코드 공간 (현재 병목) — 1바이트의 벽

**하드 리밋**: 코드 `c` → `V = (c & 0xF0) - 0x20`
- 상위니블 0x0, 0x1 → V 음수 → **사용 불가**
- 유효 상위니블 = 0x2~0xF (14행) × 16열 = **224칸이 절대 한계**

**그중 실제 안전한 코드**:
- 영어가 ASCII(0x21~0x7E) 대부분 사용
- 0x80+ 는 매크로/제어로 119종 사용 (0xC1=이름 3517회, 0xC2=아이템 611회 등)
- **전체 맵 기준 완전 안전 = 4개** (`91 DB E3 F5`)
- **단일 맵(map6) 기준 = 72개** ← PoC는 이걸로 성공

**실전 번역 규모**: 영어 대사 25,272개 런 / 약 495,743자
→ 한글 고유 음절 **700~1000자** 필요 → **1바이트로는 절대 불가**

## 8. ✅ 2바이트 훅 — 성공 (게임 화면 확인)

**"기억의 상은 / 당신을 기억합니다" 출력 성공.** 224칸 한계 돌파.

### 🔑 결정적 발견: 게임에 이미 2바이트 문자 체계가 있다

대사 렌더러(`0x80036888`)는 그리기 전에 **텍스트 전처리 함수 `0x80036514`** 를 호출한다.
```
0x800368E8: jal 0x80036514      ; a0=0x800740EC(출력버퍼), a1=원본텍스트, a2=1
```
이 함수가 원본 텍스트를 `0x800740EC` 버퍼로 복사하며 매크로를 확장하는데,
**`(byte & 0xF0) == 0xC0` 인 바이트를 2바이트 문자로 처리**한다.

| 바이트 | 전처리 동작 |
|--------|-------------|
| `0xCF` | 2바이트 복사 (팔레트) |
| `0xCE` | 2바이트 복사 |
| `0xC1~0xC7` | 매크로(이름/아이템) → 점프테이블로 확장 |
| `< 0x08` | 1바이트 (제어) |
| 그 외 | 1바이트 |

⚠️ 이 범위 **밖** 의 마커(예 `0x92`)를 쓰면 전처리가 1바이트로 쪼개고,
뒤의 인덱스 바이트를 제어문자로 오처리 → **렌더러 도달 전에 텍스트가 깨짐**.

### 채택한 인코딩: `0xCE <n>`
- `0xCE` 는 전처리를 2바이트로 자동 통과
- `0xCE` 원래 기능(테이블조회)은 대사에서 **13회만** 사용, 인자값 `{02,03,0B,0E,21,80}` 뿐
- → **`n >= 0x90` 이면 한글**, `n < 0x90` 이면 원래 기능 (안전한 분리)

### 훅 구조 (`tools/hook3.py`)
훅 자리 `0x8007AC00`, 훅 지점 `0x80036960` 을 `j 0x8007AC00` 로 교체.
```
  addiu t0, zero, 0xCE
  bne   v1, t0, L_glyph        ; v1 != 0xCE -> 글리프
  lbu   t1, 1(s3)              ; 인덱스
  addiu t0, zero, 0x90
  sltu  t2, t1, t0
  bnez  t2, L_ce               ; n < 0x90 -> 원래 CE 기능
  j     L_hangul
L_ce:    j 0x80036968
L_glyph: j 0x80036B24
L_hangul:
  addiu t1, t1, -0x90          ; idx
  ... 프리미티브 필드 채우기 (U/V 는 음절식) ...
  U = (idx & 0x0F) * 16
  V = 144 + (idx >> 4) * 15
  move  a1, s0                 ; 등록 인자
  addiu s3, s3, 1              ; 원본이 +1 더 -> 총 2바이트 소비
  j     0x80036B88             ; ★ 원본 등록/전진 코드 재활용
```

**핵심 설계**: 등록(`jal 0x8003d2c4`)과 X전진을 훅에서 직접 하지 말고
**원본 `0x80036B88` 로 점프**해 맡긴다. (직접 jal 하면 크래시)
원본이 `[s3]`(=인덱스바이트)로 폭을 조회하므로,
**폭테이블 `0x6471C + 인덱스바이트` = 13** 으로 패치 → X 전진 = 14px.

## 8-B. ⚠️ 훅 작성 시 함정 (전부 실제로 겪음)

### 1. 딜레이 슬롯 — 분기 다음 명령은 항상 실행된다
`0x80036960` 을 `j 훅` 으로 바꿔도 `0x80036964`(`addiu v0,zero,0x20`)는 **여전히 실행**된다.
→ 훅 진입 시 이미 `v0 = 0x20`. `v0` 가 0xCE 라고 가정하면 안 됨.
→ 훅에서 `t0` 에 0xCE 를 직접 로드해 비교할 것.
(글리프 경로 `0x80036B24` 는 `v0 == 0x20` 을 공백 판별에 쓰므로 이 값이 필요하기도 하다)

### 2. 🔴 로드 지연 슬롯 (R3000A) — 가장 찾기 어려웠던 버그
`lw`/`lhu`/`lbu` **바로 다음 명령**에서 그 레지스터를 읽으면 **갱신 전 값**을 읽는다.
```
  lhu   v1, -0x4bac(v0)     ; v1 = [0x8007B454]
  addiu v1, v1, 0x7f86      ; ✗ 아직 갱신 전! 이전 v1 값이 쓰임
```
증상: CLUT 에 쓰레기값(0x0006) → 한글이 **아예 안 보임** (크래시도 없음).
디버거에서 `v1 = 0x64810006` 이 단서였다 (0x64810006 - 0x7F86 = 0x64808080 = 직전 color 값).

원본 코드는 사이에 2명령을 끼워 이를 회피한다:
```
0x80036B74: lhu   v1, -0x4bac(v0)
0x80036B78: addiu v0, zero, 0xf     ; 지연 흡수
0x80036B7C: sh    v0, 0x12(s0)      ; 지연 흡수
0x80036B80: addiu v1, v1, 0x7f86    ; 이제 안전
```
→ 훅에서도 로드 후 `nop` 을 넣을 것.

### 3. 텍스트 인코딩: `0x00` 은 문자열 종료
- 음절 인덱스 `0x00` 금지 (`0xCE 0x00` → 종료로 오인)
- **끝에 `0xCF 00`(팔레트 복귀)을 두면 안 됨**:
  전처리가 `0xCF` 를 2바이트로 처리하며 뒤의 `0x00`(종료마커)을 인자로 삼켜
  문자열 끝을 못 찾고 **버퍼 오버런** → 대사 통째로 안 나옴.

### 4. 프리미티브 등록은 직접 하지 말 것
훅에서 `jal 0x8003d2c4` 를 직접 호출하면 크래시(훅 중간에서 실행 정지).
원본 `0x80036B88` 로 점프해 등록/전진을 맡기는 것이 안전.

## 8-C. 음절 폰트 배치 (현재)

ASCII 폰트 페이지 `(896,0)` 256×256 안의 **빈 영역** 사용:
- V 0~143 (코드 0x20~0xAF): 기존 ASCII/매크로 글리프
- **V 144~255: 완전히 비어 있음** → 음절 배치
- `U = (idx & 0x0F) * 16`, `V = 144 + (idx >> 4) * 15`
- 16열 × 7행 = **약 110 음절** (tpage 전환 불필요)

⚠️ 실전(700~1000 음절)엔 부족 → **tpage 전환**으로 다중 페이지 필요 (다음 작업)

## 8-D. ✅ 다중 폰트 페이지 (tpage 전환) — 성공

**Galmuri11 12×12 픽셀폰트 + tpage 전환으로 224 음절.** 게임 화면 검증 완료
(한글 출력 + 메뉴 숫자 정상).

### 폰트: Galmuri11 (픽셀폰트)
- `size=12` 에서 **안티앨리어싱 0%**, 잉크높이 11px, 글자폭 12px — 픽셀 그리드에 완벽 정렬
- 완성형 한글 11,172자 전부 지원
- Unifont 축소본보다 훨씬 선명 → **채택**
- 글리프 12×12, 잉크 = **니블값 3** (ASCII 폰트가 안 쓰는 값)

### VRAM / TIM 지도 (⚠️ 매우 중요)

| TIM | VRAM | tpage | 내용 | 한글 사용 |
|---|---|---|---|---|
| `0x7F40` | (896,0) 256×256 | `0x000E` | ASCII 폰트 | **V144~255** (페이지1, 144칸) |
| `0x11218` | (960,256) 256×256 | `0x001F` | V0~79 개발메모 / V80~143 빈칸 / **V144~255 숫자폰트** | **V80~143** (페이지2, 80칸)<br>**V0~79** (페이지3, 96칸) |
| `0x18E0` | (896,256) | `0x001E` | UI 폰트 (니블값 3 을 2072px 사용!) | ✗ |

**총 320 음절** (페이지1 144 + 페이지2 80 + 페이지3 96)

### 🔴 절대 하지 말 것 (전부 실제로 겪음)

1. **TIM `0x11218` 의 V144~255 를 건드리지 말 것 — 숫자 폰트다.**
   지우거나 VRAM 좌표를 옮기면 **메뉴의 HP/MP/G/TIME 숫자가 통째로 사라진다.**
   (겉보기엔 "개발용 디버그 폰트"처럼 보이지만 — 일본어 메모 "ここは常駐エフェクトが入る予定"가 함께 있음 — 실제로는 게임이 쓰는 폰트다)
   쓸 수 있는 건 **V80~143 (64px) 뿐**.

2. **팔레트를 통째로 덮지 말 것.** row7,8,9 는 원본에 색이 꽉 차 있고 게임 UI 가 쓴다.
   → **한글 잉크값(3) 한 칸만** 수정. 나머지 15색은 원본 보존.

3. **다른 팔레트의 값3 을 투명화하지 말 것.**
   UI 폰트 TIM(`0x18E0`)이 값3 을 2072px 쓰고 있어 숫자가 사라진다.
   그리고 투명화는 **애초에 불필요**: 한글 글리프는 V144~255 / V80~143 에만 있고
   영어 대사는 U/V 가 ASCII 영역(V0~143)만 가리키므로 한글이 영어에 나타날 수 없다.

### 중국어 패치 참고 (CN PROT 분석)
- CN 의 TIM `0x11218` 은 **US와 100% 동일** (숫자 폰트라 안 건드림)
- CN 은 대신 **ASCII 폰트 TIM(`0x7F40`)을 68.2% 채움** (US 는 7.4%)
  → 영어를 포기하고 그 페이지를 한자로 도배
- CN 파일0 의 TIM 목록/크기는 US 와 완전히 같음 → **TIM 추가 없이** 기존 것만 재활용

### 인코딩 (팔레트 코드를 페이지 선택으로 겸용)
```
0xCF 01  → 팔레트 row7 + 페이지1 (tpage 0x000E, V = 144 + (idx>>4)*12)
0xCF 02  → 팔레트 row8 + 페이지2 (tpage 0x001F, V =  80 + (idx>>4)*12)
0xCF 03  → 팔레트 row9 + 페이지3 (tpage 0x001F, V =   0 + (idx>>4)*12)
0xCE <n> → 음절, idx = n - 0x90,  U = (idx & 0x0F) * 16
```

⚠️ **페이지 전환 코드는 2바이트를 먹는다.** 음절을 페이지에 마구 흩으면
인코딩이 원문보다 길어져 삽입이 안 된다.
→ **빈도순으로 배치해 전환을 최소화**할 것.
훅이 이미 `[0x8007B454]`(팔레트 변수)를 CLUT 계산에 읽으므로 **같은 값으로 페이지도 판정**.

### tpage 전환 (페이지2)
프리미티브에 tpage 필드가 없으므로 **DR_TPAGE(GP0 0xE1)** 를 큐에 삽입.
등록 함수 `0x8003D2C4` 는 **LIFO**(나중 등록이 먼저 실행)이므로:
```
등록 순서: 복원tpage(0x000E) → sprite → 설정tpage(0x001F)
실행 순서: 설정(0x001F) → sprite → 복원(0x000E)   ✓
```
- DR_TPAGE 프리미티브 (8B): `+0 tag=0x01000000`, `+4 cmd=0xE1000000|tpage`
- s0 전진 = 0x24 (sprite 20 + tpage 8 + tpage 8)
- 검증: 한 대사 안에서 `페이지2 → 페이지1 → 페이지2` 3회 전환 정상 출력

### 🔴 MIPS 함정 추가: LUI/오프셋 부호확장
`lw/lhu/lbu/sw/sh/sb` 의 offset 은 **부호있는 16비트**다.
하위16비트가 `0x8000` 이상이면 음수가 되므로 **LUI 값에 +1** 해야 한다.
```
0x8007B454 →  LUI 0x8007 ; LHU 0xB454  →  0x8006B454  ✗ (틀림!)
              LUI 0x8008 ; LHU 0xB454  →  0x8007B454  ✓
```
증상: 팔레트 변수를 엉뚱한 주소에서 읽어 CLUT/페이지 판정이 전부 틀어짐 → 한글이 안 보임.
→ `mips.py` 의 **`hi16()` / `lo16()`** 헬퍼를 쓸 것.
